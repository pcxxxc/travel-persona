# 旅格 Travel Persona

## 基于旅游人格画像的 AI 目的地决策与手账行程系统

> TRAE AI 创造大赛参赛作品 · 生活娱乐赛道 · 2026
> 
> 内核：情绪是入口，旅行是出口。

---

## 一、产品定位：一句话

**别人帮你决定「怎么玩」，旅格帮你找到「该去哪」。**

旅格不是一个旅游攻略生成器。它是一个基于用户旅游人格画像的智能目的地决策 Agent。它的核心任务是：在用户还说不清自己想要什么的时候，接住那股模糊的情绪，把它翻译成一个可行动的目的地选择。

市面上的旅行工具都在假设你已经想清楚了——你选好城市，它帮你做攻略。旅格的能力恰恰在决策链条的更上游：它帮助你回答「我到底适合去哪」。

---

## 二、这个产品为什么值得做

### 2.1 一个被反复扼杀的念头

年轻人想旅行时，真正的难题从来不是「哪里好玩」。而是：

- 加班到深夜，脑子里冒出一句「好想出去走走」，打开手机面对几千条攻略，十分钟后默默关掉。
- 假期快到了，在小红书、抖音、携程之间反复横跳，越查越累，最后哪都没去。
- 那股想喘口气的冲动，被信息过载白白耗尽了。

那个想善待自己的微小念头，没有被任何一款产品接住过。

### 2.2 现有工具的盲区

| 工具 | 擅长 | 不擅长 |
| --- | --- | --- |
| 小红书 / 抖音 | 种草和碎片攻略 | 不问「你想要什么」，用别人的选择代替你的 |
| 携程 / 飞猪 | 机票、酒店、景点预订 | 在你还不知道去哪的时候，这些都用不上 |
| 通用 AI 旅行助手 | 给定目的地后生成行程 | 同样需要你先选好城市，缺少「选择前」的判断力 |

所有现有工具都默认用户是理性的、已经想清楚的。而那个累到想不清楚的人，恰恰没有人接住。结果不仅是丢失一次旅行的机会——更是一种自我关怀的流失。

### 2.3 情绪才是真正的入口

这一代年轻人旅行的决策驱动力，正在从「理性筛选」（几天、多少钱、有哪些景点）转向「情绪驱动」（我累了、想逃、想放空、需要新鲜感）。但市场上的产品形态还没有跟上这种变化。

旅格做的事情，就是承认并接住这种变化：**情绪是入口，旅行是出口。**

## 三、目标用户

核心用户不是「喜欢旅行的人」，而是**被一股突如其来的逃离欲击中、却无处安放的人**。定义他们的不是年龄，而是一个具体的心理瞬间。

### 典型用户画像

- **累到临界的上班族**：加班后的深夜瘫在床上刷手机，想消失几天，但说不清要去哪。
- **被生活磨平的人**：一个重复到麻木的周三下午，需要抓住点什么，让自己感觉还活着。
- **决定了但卡住的人**：假期快到，开了十个 App 又关掉，越查越累。
- **情绪过境后的人**：那阵想逃的冲动过去，几周后又想起，回来看看「那个我还在吗」。
- **学生党 / 自由职业者**：预算有限、时间灵活，但选择困难。
- **内容创作者 / 摄影爱好者**：更在意城市氛围、空间质感、灵感密度。
- **潜在数字游民**：不只想旅行，更想判断一座城市是否适合长住和工作。

### 核心使用场景的独特性

关键洞察：**用户打开旅格时，往往还没决定要旅行。** 他们是被一种情绪驱使过来的，旅行只是这股情绪可能的出口之一。旅格的产品任务，是在信息过载吞噬这个念头之前，用最少的时间（1 分钟到 3 分钟）完成一次从「我不知道我想要什么」到「这就是我现在需要的」的转化。

## 四、产品架构：三种入口，层层递进

旅格设计了三种不同深度的分析入口，用户可以根据自己的意愿和时间灵活选择。

### 模式一：极速模式（1 分钟）

一屏一个情绪选择加一个追问即可完成。

用户只需做两件事：
1. 从 7 种情绪中选择一个最能描述当下状态的（放空、逃离压力、找灵感、拍照出片、社交、独处整理、试住城市）
2. 回答一个漏斗式追问（如：你希望空间给你什么感觉？）

可选展开高级设置（旅行天数、预算），但不是必选项。点击「开始分析」后，系统基于情绪（第一驱动力）直接推断人格并给出推荐。

适合场景：冲动型决策、「先看看再说」、碎片时间随手尝试。

### 模式二：10 题版（约 3 分钟）

10 道题目覆盖情绪、空间偏好、预算、时间、旅行经历、厌恶项、节奏、优先级、风险容忍度、数字游民意向。数据更丰富，分析更准确。

适合场景：认真考虑一次旅行、想获得针对性更强的推荐。

### 模式三：20 题版（约 5 分钟）

20 道题目提供最完整的数据采集，包括当前位置、偏好城市类型等额外维度，输出结果更精确。

适合场景：深度自我探索、愿意为更精准的推荐投入时间。

### 入口设计哲学

三种模式的入口是并列呈现的，不做层级隐藏。用户首先看到的就是极速模式的核心交互（情绪选择），底部收起的是另外两个模式的卡片入口。这样设计的原因是：不要在产品入口就增加决策成本——大多数用户在打开旅格时本身就已经很疲惫了。

### 问卷的渐进式采集与中断恢复机制

旅格的问卷设计遵循一个关键原则：**随时随地可以结束，也随时随地可以继续。**

#### 为什么需要这个机制

传统问卷强迫用户一次性完成所有题目，这在旅格的场景下尤其不妥。用户打开旅格时往往处于疲惫、犹豫、被打扰的状态——他们可能在地铁上、在会议间隙、在睡前刷手机。强制填完 20 题只会加速他们关掉页面。更糟的是，那些已经填了 8 题就快完成的人，一旦中断就得重来，而他们很可能不会再来了。

#### 设计细则

- **每道题即时保存**：每次用户选择或输入后，答案即刻写入 localStorage。没有任何显式的「保存」按钮——保存是无声的。
- **随时可离开**：10 题版和 20 题版的每个步骤顶部有一个不显眼的「先到这」按钮，点了就结束，不追问、不挽留。
- **回来时主动提醒**：用户下次打开旅格时，如果在 localStorage 中检测到未完成的答卷，在入口页顶部展示一个温和的横幅——「你上次的回答还在，要继续吗？」——附带已完成的进度（如 5/10）。
- **允许丢弃重来**：横幅同时提供「重新开始」和「先不管」两个选项，控制权始终在用户手里。
- **已回答的题目计算即时匹配分**：即使用户在 10 题版只答了 6 题就中断，系统也允许以已有数据立即生成推荐（置信度会标注为「部分数据」），不让用户觉得「白填了」。
- **后续补充触发人格更新**：用户某天回来补完全部题目后，系统基于新数据重新计算 PersonaScore 并更新推荐。这个过程本身就是一个微型的「人格演化」。

#### 对粘性的意义

中断恢复机制在产品层面创造了三个粘性钩子：

- **蔡格尼克效应**——未完成的事在记忆中更鲜活，用户会惦记着「那几道没答完的题」。
- **沉没成本温和绑定**——已经填了 8 题，回来补完只差 2 题，比从零开始填 10 题容易得多。
- **二次触达的理由**——「你的答案还在」是一个自然且不骚扰的召回理由，比推送通知高明得多。

## 五、系统架构：两阶段设计

旅格的核心架构遵循一个明确的原则——什么该用确定性流程，什么该用 Agent。

### 第一阶段：推荐 Workflow（确定性流程）

从用户输入到城市推荐的完整链路是**无动态规划的确定性流程**。数据全部已知，执行顺序固定，不依赖 Agent 做决策。

```
用户回答
      ↓
维度提取  —— LLM 输出离散档位的维度增量
      ↓
PersonaScore（六维人格分）
      ↓
人格判定  —— 规则推导（派生展示，非输入）
      ↓
六维评分  —— 加权欧氏距离匹配
      ↓
候选城市 Top10
      ↓
天气过滤  —— 极端天气降权
      ↓
Top3 排序
      ↓
LLM 生成推荐理由 —— 引用原话 + 诚实提醒
      ↓
推荐完成 → 进入第二阶段
```

### 第二阶段：陪伴 Agent（动态决策）

推荐完成后，Agent 才介入。它有长期状态、持续观察、动态决策能力——这正是 Agent 定义成立的地方。详见第十一节。

### 为什么这样划分

| 环节 | 实现方式 | 需要 LLM？ | 需要动态决策？ |
| --- | --- | --- | --- |
| 维度提取 | 单次 LLM | 是 | 否 |
| 人格判定 | 规则推导 | 否 | 否 |
| 六维评分 | 本地算法 | 否 | 否 |
| 行程求解 | 本地规则 | 否 | 否 |
| 理由/行程润色 | 单次 LLM | 是 | 否 |
| 旅行中陪伴 | Agent（观察-推理-行动） | 是 | 是 |

两阶段解耦带来的好处：Agent 模块完全独立，即使它故障，第一阶段的推荐链路照常工作。

## 六、人格维度：从六种人格到底层六维

旅格的底层不直接使用人格标签做匹配，而是用一套**六维连续数值体系（PersonaScore）**来刻画用户的旅行倾向。

### 六维人格分

```
{
  "freedom": 0.83,   自由探索倾向
  "social":  0.21,   社交/热闹倾向
  "explore": 0.72,   深度探索/小众倾向
  "nature":  0.65,   自然/户外倾向
  "pace":    0.30,   快节奏/高效率倾向
  "budget":  0.55    高消费/品质倾向
}
```

每维取值 0~1。六种人格名（松弛漫游者、自然疗愈者等）是由这组分值派生出的展示标签，不参与计算。

### 六种旅游人格

| 人格 | 核心特征 | 典型场景 | 匹配城市类型 |
| --- | --- | --- | --- |
| 松弛城市漫游者 | 无固定目的地，感受一座城的呼吸而非打卡 | 老街、咖啡书店、散步、发呆 | 慢节奏、有旧街区、咖啡文化发达 |
| 自然疗愈逃离者 | 在山海森林星空之间找回内心平静 | 山海、徒步、低开发自然区 | 自然占比高、商业化低、安静 |
| 烟火气探索者 | 用味蕾和脚步认识一座城市 | 夜市、本地餐馆、菜市场、老社区 | 本地生活浓度高、美食密集、市井 |
| 高效打卡收集者 | 在有限时间里最大化体验 | 地标建筑、博物馆、紧凑路线 | 交通便利、景点密集、节奏偏快 |
| 灵感采集型创作者 | 采集建筑线条、街区色彩、光影变化 | 艺术街区、建筑、展览、设计店 | 审美密度高、有艺术区、出片率高 |
| 数字游民试居者 | 评估一座城市是否适合生活和工作 | 咖啡馆办公、共享空间、生活区探索 | 数字游民友好、生活成本适中、短住便利 |

### 维度提取：可解释的离散档位

为保证每一维度的数值变化都可追溯，项目不采用黑盒连续值输出，而是使用**离散档位映射**。

对于结构化选项（如「推开门你想看到什么？」→ 海），预设离散增量：
```
海 → nature +0.3, social -0.3, pace -0.3
老街 → social +0.3, explore +0.1, nature -0.3
咖啡馆 → pace -0.3, freedom +0.3
山 → explore +0.3, nature +0.3, freedom +0.1
```

对于自由文本，LLM 也只被允许在固定档位中选择：

| 档位 | 语义 |
| --- | --- |
| +0.5 | 强正向信号 |
| +0.3 | 中正向信号 |
| +0.1 | 弱正向信号 |
| 0 | 无信号 |
| -0.1 | 弱负向信号 |
| -0.3 | 中负向信号 |
| -0.5 | 强负向信号 |

最终维度分 = 0.5 基准 + 各题增量之和，裁剪到 [0, 1]。每一步增量都对应明确的语义判断，而非黑盒小数。

### 人格推断的权重递进

即使只有一条情绪信号，系统也能给出有意义的推断；数据越多，画像越细。

1. 情绪目标（最高权重）——「为什么想旅行」是第一驱动力
2. 空间偏好——作为情绪信号的辅助细化
3. 旅行时长——短途偏高能高效，长途偏慢节奏深度
4. 数字游民意向——如有明确意向则大幅提升试居型权重
5. 预算——低预算偏性价比探索，高预算偏品质体验

## 七、匹配算法：城市与人的六维对话

旅格的推荐不是随机的，也不是基于热度排名的。每座城市在数据库中被拆解为与用户人格**同构的六维向量**，算法计算两者之间的加权欧氏距离。

### 六维加权评分

```
distance = sqrt( Σ wᵢ × (userᵢ − cityᵢ)² )
matchScore = round( (1 − distance / maxDist) × 100 )
```

| 维度 | 权重 | 依据 |
| --- | --- | --- |
| nature | 0.25 | 最影响目的地大类（山海 vs 都市） |
| pace | 0.20 | 决定城市气质匹配度 |
| social | 0.20 | 情绪驱动用户的核心诉求 |
| budget | 0.15 | 现实约束 |
| explore | 0.12 | 影响小众 vs 热门 |
| freedom | 0.08 | 更多影响行程安排而非选城市 |

### 城市画像数据库

当前覆盖 20 个中国热门目的地城市，每座城市以六维向量表征。城市标注采用**双人独立标注 + 交叉校准**的流程：两名标注者独立为每城六维打分，差值超过 0.2 的维度进入协商，并参考公开平台（马蜂窝/小红书）的城市标签词频做客观锚定。这让「大理 pace = 0.2」有据可依，而非主观拍板。

每座城市还包含精选真实 POI（每城 6-10 个），带 zone（区域）、openHours（营业时间）、indoor（室内/室外）字段，为后续的行程约束求解提供数据基础。

### 推荐链路：Top10 + 天气过滤

```
全城算分 → Top10 → 查天气 → 极端天气 ×0.85 降权 → 重排 → Top3
```

如果天气 API 故障，跳过天气过滤直接推荐，不影响核心功能。

### 推荐输出不只有分数

每个推荐城市不仅给出匹配度百分比，还由 LLM 生成**引用用户原话的个性化推荐理由**，以及诚实的风险提醒：

- 为什么推荐你——算法理由文本
- 你可能需要注意什么——诚实列出不适合之处和潜在风险
- 适合待几天、符合预算吗、适合普通旅行还是短住试居

这是一个会劝退你的推荐工具，而不是一味讨好。

## 八、行程生成：规则求解 + LLM 润色

行程不是「全靠 LLM 编」，而是先用代码解出满足约束的骨架，再让 LLM 润色成有温度的文字。

### 约束求解层（本地代码）

```
输入：Top1 城市的 POI 列表 + 天数 + 天气 + personaScore

1. 按 zone 聚类     → 同区域 POI 安排在同一天，减少折返
2. 营业时间过滤    → 排除 openHours 冲突
3. 雨天替换        → 当天 weather=rain，把 outdoor POI 替换为 indoor 备选
4. 节奏适配        → pace 低则每天少排 POI（留白），pace 高则排满
      ↓
结构化行程骨架（哪天、哪些点、什么顺序）
```

### LLM 润色层

将骨架写成有画面感的每日描述，赋予主题词和实用贴士。这样行程的「合理性」由代码保证（不折返、不撞营业时间、雨天有替代），「温度」由 LLM 保证。

六种人格各有不同的行程模板风格：

- 松弛漫游者：少景点、重步行、咖啡馆、老街，不设严密的点位路线
- 自然疗愈者：低强度自然路线、少商业化、重放空、早起看日出
- 烟火探索者：早市→老社区→本地餐馆→夜市，以「吃」为线索串联
- 高效打卡者：路线紧凑、交通优化、减少折返，每个点位分配合理时间
- 灵感采集者：建筑→展览→街区→机位→材质观察，以「记录」串联
- 数字游民试居者：工作空间→生活配套→租住区域→城市日常节奏探索

## 九、旅行手账与情绪迁徙地图

### 旅行手账

最终输出不是冷冰冰的表格，而是一份可打印的旅行手账页面，包含：

- 旅行人格卡片
- 推荐城市卡片与匹配分
- 每日行程时间轴与当日关键词（如「慢走、老街、咖啡、日落」）
- 情绪标签（如「松弛 / 安静 / 被城市接住」）
- 每日引导性记录问题（如「今天哪个空间让你最想停下来？」）
- 自由书写区域
- 可分享的手账封面

手账将旅格从「决策工具」延伸为「旅行记忆载体」。

### 情绪迁徙地图

用户每做一次旅行人格测试，系统就记录一个情绪节点。用得越久，这些节点连成一条线：

> 过去半年，你的心从「想躲起来」到「想扎进人群」再到「想登高望远」，如何流动。

这不是为了标注你去过哪里，而是为了让你看见自己正在如何变化。这份记录只存在用户的设备上。

## 十、人格演化：从静态标签到成长闭环

前面的 PersonaScore 是静态的——测一次得一个分，推荐完就结束。但人是会变的。**人格演化（Persona Evolution）** 让数值随每次真实旅行更新，使用户不是被测量一次，而是在持续成长。

### 闭环结构

```
PersonaScore (social: 0.20)
      ↓
推荐大理 + 行程
      ↓
真实旅行（和民宿老板聊了很久）
      ↓
旅行反思："今天和谁聊得开心吗？"
      ↓ 从回答中提取情绪/维度信号
PersonaScore 更新 (social: 0.20 → 0.35)
      ↓
下一次推荐，基于"成长了一点的你"
```

### 更新机制

从旅行反思的回答中提取维度信号（复用离散档位），对每个有信号的维度做**小步更新**，防止剧烈跳变：

```
newScore = oldScore + α × delta    (α ≈ 0.3，阻尼系数)
```

α 阻尼保证人格「缓慢演化」而非「一次旅行就剧变」，符合真实成长节奏。每次变化记录一条 EvolutionLog：哪个维度、从多少到多少、触发事件是什么、用户当时说了什么。

### 人格演化的意义

- **粘性的终极来源**：用户回来不是为了找下一个城市，而是为了看看「这次旅行把我变成了什么样」。
- **两条线叠加**：情绪迁徙地图记录「你想去哪」（状态轨迹），人格演化记录「你变成了谁」（成长轨迹）。两条线一起构成完整的自我档案。
- **数据闭环**：PersonaScore 不再是输入端的一次性测量，而是贯穿用户生命周期、持续被旅行经历塑造的活数据。

> 当前阶段：初赛不做演化，PersonaScore 静态即可。复赛实现演化闭环——这是从「推荐工具」升华为「成长伙伴」的关键一步。

## 十一、Travel Companion Agent：推荐之后的整个旅行生命周期

这是项目中唯一真正的 Agent 模块。它不碰「推荐哪个城市」（那是 Workflow 的事），只负责推荐完成之后的三个阶段。

### 为什么这里才需要 Agent

推荐是确定性的——问什么答什么，数据已知，流程固定。但「旅行中」和「长期情绪」是动态的、不可预知的——用户什么时候说「下雨了」、半年后情绪怎么变，无法预先编排。这正是 Agent 的用武之地：持续观察状态、根据新信息推理、采取行动、再观察。

### 三种能力（同一个 Agent，三种切入）

**能力一：旅行中实时调整（观察-推理-行动闭环）**

用户说「今天下雨」。Agent 读 TravelState（大理 / Day2 / rain / 能量低），推理出原行程下午的洱海骑行（室外）受雨影响且用户能量偏低，调用天气工具确认后从城市库检索室内 POI，把下午改成古城咖啡馆，骑行挪到放晴的 Day3。这是完整的 ReAct 式循环。

**能力二：长期情绪观察（基于情绪时间线）**

Agent 周期性扫描情绪序列。发现用户连续多次选择低刺激环境（疲惫→海、焦虑→海、迷茫→海），形成假设：用户可能长期处于高压。主动输出观察：「这半年你一直在找安静的地方。今年暑假，也许你需要的不是又一次打卡，而是一次真正的放空。」Agent 从「推荐器」变成「观察者+建议者」。

**能力三：旅行反思（回归自我观察主线）**

旅行中或结束时，Agent 递出温柔的反思问题：

> 「今天最放松的瞬间是什么？」  
> 「哪个地方最让你意外？」  
> 「你觉得，自己和出发前有什么不一样了吗？」

关键纪律：主语始终是用户自己。旅格只递问题、当镜子，不当教练、不布置任务。反思的产出是第十节人格演化的输入。

### 落地优先级

- 初赛：不做 Agent，只做第一阶段 Workflow（推荐 + 行程 + 情绪地图）。
- 复赛：实现能力一（旅行中调整，最完整的 Agent 闭环）为主，能力二和三作为展示亮点。

## 十二、双层产品架构

旅格的产品结构由两层组成，既保证免费可及性，也预留增值空间。

### 第一层：算法匹配（免费）

本地运行的多维人格推断 + 城市匹配算法。用户输入数据后，推荐链路在浏览器端完成计算，仅 LLM 调用（维度提取、理由生成、行程润色）经过后端。这一层是产品的核心体验闭环——从情绪到人格到城市推荐，始终免费。

### 第二层：AI Agent 对话（高级服务 / 积分兑换）

接入大语言模型，提供：
- 旅行中实时观察与动态行程调整
- 长期情绪观察与主动建议
- 旅行反思引导与人格演化更新

两层架构的设计意图：算法层解决 80% 的基础需求，让大多数用户无需付费即可完成一趟从困惑到行动的全流程。Agent 层提供更细腻的陪伴体验，适合愿意投入的用户。

## 十三、异常处理与降级策略

任何外部依赖失败，核心推荐都能降级跑通，现场绝不白屏。

| 故障点 | 降级方案 |
| --- | --- |
| 维度提取 LLM 失败 | 退回纯选项打分，不依赖自由文本 |
| 理由生成失败 | 使用预设高质量模板理由兜底 |
| 行程润色失败 | 直接展示规则求解的骨架（虽朴素但可用） |
| 天气 API 失败 | 跳过天气过滤，行程天气字段显示「暂无」 |
| Companion Agent 故障 | 不影响推荐链路，陪伴功能优雅降级 |
| localStorage 不可用 | 情绪地图隐藏，核心推荐不受影响 |

## 十四、市场竞争位置

旅格不是要取代携程或小红书，而是补充它们没有覆盖的那一环节。

```
携程 / 飞猪：预订工具（决策链最下游）
小红书 / 抖音：种草平台（中游，信息供给）
旅格：目的地决策（最上游，帮你先想清楚）
```

旅格做的，是决策链最上游的「方向收敛」——把无限的信息噪声收束成一个对你此刻状态真正合适的选项。用户在旅格完成决策后，仍会回到携程买票、去小红书写笔记、打开地图导航。旅格不抢这些工具的生意，而是在它们覆盖不到的空白处创造价值。

## 十五、技术实现与成本

### 技术架构

- **前端**：HTML5 + CSS3 + Vanilla JavaScript，纯原生实现，无框架依赖
- **本地算法**：六维加权评分 + 行程约束求解，在浏览器端完成
- **后端**：Node.js server，负责 LLM API 代理（维度提取、理由生成、行程润色）和天气 API 代理
- **LLM 调用策略**：提取用 Haiku（成本低），理由/行程用 Sonnet（质量优先）
- **响应式设计**：适配桌面端和移动端
- **流式输出**：推荐理由和行程润色采用 SSE 流式返回，减少等待感

### 成本估算

单用户单次 LLM 调用约 4-5K tokens（维度提取 0.5K + 理由 1K + 行程润色 2K + 偶发洞察 1K）。100 用户/天 × 1.5 次 ≈ 750K tokens/天，日成本个位数美元。

| 项 | 策略 |
| --- | --- |
| 天气 API | 后端缓存 6 小时，同城共享 |
| IP 限流 | 每分钟 10 次 |
| 模型选择 | 提取用 Haiku，理由/行程用 Sonnet |
| 部署 | 阿里云 Ubuntu + PM2，单实例 |

## 十七、项目的三层价值

**效率价值**：把一次原本需要刷一小时攻略、最后往往无果的旅行决策，压缩成一次 1-3 分钟的轻量交互。不给你更多选择，而是帮你把选择收敛成一个真正适合此刻的答案。

**情感价值**：旅格用情绪的语言接住那个说不清自己想要什么的人，帮他确认「我现在真正需要的是什么」。哪怕最后没有出发，他也完成了一次对自己的认真对待。它不替代休息、不假装治愈人生，只是不让那个想对自己好一点的念头，被白白扼杀。

**社会价值**：当一个人愿意为自己的情绪停下来想一想「我需要什么」，这本身就是一种珍贵的自我关怀。旅格降低的，不只是旅行决策的成本，更是年轻人善待自己的门槛。
