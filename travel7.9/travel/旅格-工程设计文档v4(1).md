# 旅格 Travel Persona · 工程设计文档(v4)

> 修订说明(v4 基于 v3):
> ① 能力三"人生实验"收敛为"Travel Reflection 旅行反思",回归自我观察主线,不做成长教练;
> ② 新增第 14 节 Persona Evolution(人格演化模型),让人格随每次旅行更新,形成"测量→推荐→旅行→反思→人格成长"的完整闭环——用户不是被测量,而是在成长。
>
> (v3 要点保留:维度提取离散化、城市标注交叉校准、行程约束求解、两阶段架构。)

---

## 1. 产品目标

旅格是基于 Claude API 的旅行决策网站。用户通过投射式对话表达此刻情绪,系统推断旅行人格,推荐最契合的城市并给出诚实理由,生成可调整行程,沉淀情绪轨迹,并在旅行中持续陪伴。

**内核:情绪是入口,旅行是出口。**

**架构总原则(v3 确立):**
- **推荐链路 = 确定性 Workflow**(稳定、可解释、快)
- **陪伴模块 = 真 Agent**(长期状态、持续观察、动态决策)
- Agent 不参与"推荐哪个城市",只参与"推荐之后怎么办"

---

## 2. 用户链路(核心决策链路)

```
用户回答(对话)
      ↓
[1] 维度提取    —— LLM 输出离散档位的维度增量(见 5.1)
      ↓
personaScore { freedom, social, explore, nature, pace, budget }
      ↓
[2] 人格判定    —— 规则:维度分 → 6 种人格(派生展示,非输入)
      ↓
[3] 六维评分    —— 加权欧氏距离 → 匹配分
      ↓
候选城市 Top10
      ↓
[4] 天气过滤    —— 查天气,极端天气降权
      ↓
Top3 排序
      ↓
[5] LLM 生成推荐理由 —— 引用原话 + 诚实提醒
      ↓
推荐完成 ──────→ 进入第二阶段:Travel Companion Agent(第 13 节)
```

[1]-[5] 全程无动态规划,是确定性流程。Agent 在推荐完成之后才介入。

## 3. 系统架构(两阶段)

```
═══ 第一阶段:推荐 Workflow ═══
前端
  ├── 对话测试 / 推荐展示 / 行程页
  ├── 六维评分(本地 JS,纯函数)
  └── 行程约束求解(本地 JS,见 5.5)
        │
后端 server.js
  ├── /api/extract    维度提取(LLM,离散输出)
  ├── /api/reason     推荐理由(LLM,流式)
  ├── /api/itinerary  行程润色(LLM,流式)
  └── /api/weather    天气代理 + 缓存

═══ 第二阶段:陪伴 Agent ═══

  ├── EmotionTimeline(长期情绪序列)

```

两阶段解耦:**Agent 模块完全独立,即使它故障,第一阶段的推荐链路照常工作。**

## 4. 核心数据模型

### 4.1 PersonaScore(人格维度 · 推荐依据)
```json
{
  "freedom": 0.83,   "social": 0.21,  "explore": 0.72,
  "nature":  0.65,   "pace":   0.30,  "budget":  0.55
}
```
每维 0~1。人格名由这组分值派生,仅用于展示,不参与计算。

### 4.2 City(城市模型 · 与人格维度同构)
```json
{
  "id": "dali", "name": "大理",
  "dimensions": { "freedom":0.80, "social":0.30, "explore":0.60,
                  "nature":0.90, "pace":0.20, "budget":0.45 },
  "emotionTags": ["治愈","逃离","放空"],
  "pois": [
    { "name":"洱海生态廊道", "zone":"洱海东岸", "type":"自然",
      "openHours":"全天", "indoor":false, "note":"骑行/发呆" },
    { "name":"人民路", "zone":"古城", "type":"街区",
      "openHours":"10:00-22:00", "indoor":false, "note":"咖啡馆小店" }
  ]
}
```
POI 新增 `zone`(区域)、`openHours`(营业时间)、`indoor`(室内/室外)三个字段——这是为 5.5 的行程约束求解准备的。

### 4.3 EmotionNode(单次情绪节点)
```json
{ "date":"2026-01-15", "emotion":"想消失几天", "door":"海",
  "personaScore": {...}, "recommendedCity":"大理" }
```

### 4.4 TravelState(旅行中状态 · Agent 维护)
```json
{ "city":"大理", "day":2, "totalDays":4,
  "weather":"rain", "energy":30, "mood":"tired",
  "originalItinerary": {...}, "completedActivities": [...] }
```

### 4.5 Itinerary(行程模型)
```json
{ "city":"大理", "days":[
  { "day":1, "weather":"晴 18℃", "theme":"抵达与慢下来",
    "morning":"...", "afternoon":"...", "evening":"..." }
]}
```

## 5. 推荐算法(核心)

### 5.1 问卷 → 维度映射(可解释)

**结构化选项:预标注离散增量**
```
"推开门你想看到什么?"
  海   → nature +0.3, social -0.3, pace -0.3
  老街 → social +0.3, explore +0.1, nature -0.3
  咖啡馆 → pace -0.3, freedom +0.3
  山   → explore +0.3, nature +0.3, freedom +0.1
```

**自由文本:LLM 输出离散档位(关键改进)**

为保证提取可解释,LLM 不输出任意小数,只能在固定档位中选择:
```
允许的档位:-0.5 / -0.3 / -0.1 / 0 / +0.1 / +0.3 / +0.5
含义:    强负 / 中负 / 弱负 / 无 / 弱正 / 中正 / 强正
```
Prompt 强约束:对每个维度,LLM 必须判断"用户这句话对该维度是强/中/弱/无信号",输出对应档位。例:"累得不想说话" → social: -0.3(中等负向社交信号)。

这样,提取过程和评分过程一样可解释:每个增量都对应一个明确的语义档位,而非黑盒小数。

**最终:** 维度 = 0.5 基准 + 各题增量之和,裁剪到 [0,1]。

### 5.2 六维加权评分
```
distance = sqrt( Σ wᵢ × (userᵢ − cityᵢ)² )
matchScore = round( (1 − distance / maxDist) × 100 )
```

| 维度 | 权重 | 依据 |
|------|------|------|
| nature | 0.25 | 最影响目的地大类 |
| pace | 0.20 | 决定城市气质匹配 |
| social | 0.20 | 情绪驱动用户核心诉求 |
| budget | 0.15 | 现实约束 |
| explore | 0.12 | 影响小众 vs 热门 |
| freedom | 0.08 | 更多影响行程,非选城市 |

权重和为 1,Demo 阶段可调,后续用"推荐采纳率"反推优化。

### 5.3 TopK + 天气过滤
```
全城算分 → Top10 → 查天气 → 极端天气 ×0.85 降权 → 重排 → Top3
```

### 5.4 LLM 生成推荐理由(`/api/reason`)
输入用户原话 + Top1 城市 + 维度对比;约束:引用用户原词、给诚实提醒、80-120字、流式。唯一"生成质量决定上限"的环节。

### 5.5 行程约束求解(关键改进:规则编排 + LLM 润色)

行程不再"全靠 LLM 编",而是先用代码解出满足约束的骨架,再让 LLM 润色成有温度的文字。

```
输入:Top1 城市的 POI 列表 + 天数 + 天气 + personaScore
      ↓
[规则求解层 · 本地代码]
  1. 按 zone 聚类      —— 同区域 POI 安排在同一天,减少折返
  2. 营业时间过滤      —— 用 openHours 排除时段冲突
  3. 雨天替换          —— 当天 weather=rain,把 indoor:false 的 POI
                          替换为 indoor:true 的备选
  4. 节奏适配          —— pace 低则每天少排 POI(留白),pace 高则排满
      ↓
结构化行程骨架(哪天、哪些点、什么顺序)
      ↓
[LLM 润色层 · /api/itinerary]
  把骨架写成有画面感的每日描述 + 主题词 + 贴士
      ↓
最终 Itinerary
```

这样行程的"合理性"由代码保证(不折返、不撞营业时间、雨天有替代),"温度"由 LLM 保证。工程味和体验兼得。

## 6. Agent / Workflow 边界

**第一阶段(推荐)= Workflow,不用 Agent、不用 ReAct。** 数据全部已知,执行顺序固定,无动态规划需求。论证见下表。

| 环节 | 实现 | LLM | 动态决策 |
|------|------|-----|---------|
| 维度提取 | 单次 LLM | 是 | 否 |
| 人格判定 | 规则 | 否 | 否 |
| 六维评分 | 本地算法 | 否 | 否 |
| 行程求解 | 本地规则 | 否 | 否 |
| 理由/行程润色 | 单次 LLM | 是 | 否 |

**第二阶段(陪伴)= 真 Agent**,详见第 13 节。它有长期状态、持续观察、动态决策,是 Agent 定义成立的地方。

**关于工具:** 天气是真外部 Tool;城市库是本地数据查询;情绪档案是前端状态管理(不伪装成 Agent Tool)。

## 7. 接口设计

### POST /api/extract
```
入参:{ "freeText":"累得不想说话", "currentScore":{...} }
返回:{ "delta": { "social":-0.3, "pace":-0.3 },
        "rationale": { "social":"中等负向社交信号" } }
```
返回带 rationale,让每个增量可追溯。

### POST /api/reason(流式)
```
入参:{ "personaScore":{...}, "userQuotes":[...],
        "topCity":{...City...}, "candidates":["苏州","青海湖"] }
返回:SSE → { "reason":"...", "honestNote":"..." }
```
传 personaScore 而非 persona。

### POST /api/itinerary(流式)
```
入参:{ "city":"大理", "skeleton":{...求解后的骨架...},
        "personaScore":{...}, "adjustInstruction":"第二天改轻松点" }
返回:SSE → Itinerary
```
注意:入参是已经过 5.5 规则求解的 `skeleton`,LLM 只润色不重排。

### GET /api/weather(代理 + 缓存 6h)
```
入参:?city=dali&days=7
返回:{ "daily":[ {"date","temp","condition"} ] }
```

### POST /api/companion(Agent · 见第 13 节)
```
入参:{ "travelState":{...}, "emotionTimeline":[...], "userMessage":"今天下雨" }
返回:SSE → { "observation","reasoning","action","newPlan" }
```

## 8. 状态机(对话测试终止条件)
```
COLLECTING → READY
每轮后:更新 personaScore → 算置信度(已明确赋值维度数/6)
终止(任一):置信度 ≥0.8(≥5维有信号) 或 已问满 5 轮
未终止:针对信号最弱的维度生成下一问
```
5 轮硬上限是体验底线。

## 9. 城市知识库设计

| 项目 | 方案 |
|------|------|
| 规模 | 初赛 20 城,复赛 40 城(中国热门目的地) |
| 范围 | **闭集推荐**——只在库内推荐,杜绝瞎编城市/景点 |
| 维度标注流程 | **双人独立标注 → 差值 >0.2 的维度协商校准 → 参考公开旅游标签(马蜂窝/小红书城市标签词频)做锚定** |
| POI 来源 | 每城精选 6-10 个真实地点,人工录入,含 zone/openHours/indoor |
| 维护 | 静态 JSON,版本化;复赛前一次性扩充 |

标注可信度说明:每个城市的 6 维分值由两名标注者独立打分,差异超过 0.2 的维度进入协商,并参考公开平台的城市标签词频做客观锚定(如"大理"在公开数据中"治愈/慢"标签占比高 → 佐证 pace 低)。这让"大理 pace=0.2"有据可依,而非主观拍板。

## 10. 异常处理与降级

| 故障点 | 降级 |
|--------|------|
| 维度提取 LLM 失败 | 退回纯选项打分,不依赖自由文本 |
| 理由生成失败 | 用预设高质量模板理由兜底 |
| 行程润色失败 | 直接展示规则求解的骨架(虽朴素但可用) |
| 天气 API 失败 | 跳过天气过滤,行程天气字段显示"暂无" |
| Companion Agent 失败 | 不影响推荐链路,陪伴功能优雅降级 |
| localStorage 不可用 | 情绪地图隐藏,核心推荐不受影响 |

原则:任何外部依赖失败,核心推荐都能降级跑通,现场绝不白屏。

## 13. Travel Companion Agent(复赛扩展)

这是项目唯一真正的 Agent 模块。它不碰"推荐哪个城市"(那是 Workflow 的事),只负责"推荐之后的整个旅行生命周期"。

### 13.1 为什么这里才需要 Agent

推荐是确定性的,但"旅行中"和"长期情绪"是动态的、不可预知的——用户什么时候说"下雨了"、半年后情绪怎么变,无法预先编排。这正是 Agent 的用武之地:**持续观察状态、根据新信息推理、采取行动、再观察。**

### 13.2 三种能力(同一个 Agent,三种切入)

**能力一:旅行中实时调整(观察-推理-行动闭环)**
```
用户:"今天下雨"
  观察:读 TravelState(大理 / Day2 / rain / energy 低)
  推理:原行程下午是洱海骑行(室外),受雨影响;用户能量也低
  行动:调用天气工具确认 + 从城市库检索室内 POI
  输出:把下午改成古城咖啡馆,骑行挪到放晴的 Day3
```
这是完整的 ReAct 式循环——但只在这个真正需要动态决策的场景使用,不滥用到推荐链路。

**能力二:长期情绪观察(基于 EmotionTimeline)**
```
Agent 周期性扫描情绪序列:
  [1月 疲惫→海] [3月 焦虑→海] [6月 迷茫→海]
  形成假设:用户连续多次选择低刺激环境,可能长期处于高压
  主动建议:"这半年你一直在找安静的地方。今年暑假,
            也许你需要的不是又一次打卡,而是一次真正的放空。"
```
Agent 从"推荐器"变成"观察者+建议者"。

**能力三:旅行反思(Travel Reflection · 回归自我观察主线)**
```
旅行中或结束时,Agent 递出温柔的反思问题(不指导,只发问):
    "今天最放松的瞬间是什么?"
    "哪个地方最让你意外?"
    "你觉得,自己和出发前有什么不一样了吗?"
  用户回答 → Agent 从回答中提取情绪/维度信号 → 喂给第 14 节的人格演化
```
关键:主语始终是用户自己。旅格只递问题、当镜子,不当教练、不布置任务。这守住"克制不煽情"纪律,也和"情绪→旅行→自我观察"主线统一。反思的产出是 Persona Evolution 的输入。

### 13.3 技术实现
```
新增 TravelState(见 4.4)+ 复用 EmotionTimeline
/api/companion 接收 { travelState, emotionTimeline, userMessage }
  → Planner 判断是否需要调整/建议
  → 必要时调用天气工具 / 城市库
  → 输出新计划或建议
Agent 维护状态,跨消息保持上下文。
```

### 13.4 两阶段架构的两个好处
1. **核心稳定** —— Agent 故障,推荐功能照常工作。
2. **Agent 是真 Agent** —— 长期状态 + 持续观察 + 动态决策,而非"查天气→查城市→结束"的伪 Agent。

### 13.5 落地优先级
初赛:不做 Agent,只做第一阶段 Workflow(推荐 + 行程 + 情绪地图)。
复赛:实现能力一(旅行中调整,最完整的 Agent 闭环)为主,能力二/三作为展示亮点。

## 14. Persona Evolution(人格演化模型)

这是整个产品的灵魂闭环。它解决一个根本问题:**之前的 PersonaScore 是静态的——测一次得一个分,推荐完就结束,等于把人看成固定标签。但人是会变的。**

Persona Evolution 让人格分值随每次真实旅行而更新,于是:**用户不是被测量一次,而是在持续成长。**

### 14.1 闭环结构
```
PersonaScore(social: 0.20)
      ↓
推荐大理 + 行程
      ↓
真实旅行(和民宿老板聊了很久)
      ↓
Travel Reflection(能力三):"今天和谁聊得开心吗?"
      ↓ 用户回答 → 提取信号
PersonaScore 更新(social: 0.20 → 0.35)
      ↓
下一次推荐,基于"成长了一点的你"
```

### 14.2 更新机制
```
旅行后,从 Travel Reflection 的回答中提取维度信号(复用 5.1 的离散档位)
对每个有信号的维度做小步更新(防止剧烈跳变):
    newScore = oldScore + α × delta     (α ≈ 0.3,阻尼系数)
    裁剪到 [0,1]
记录一条 EvolutionLog:{ date, dimension, from, to, triggerEvent }
```
α 阻尼保证人格"缓慢演化"而非"一次旅行就剧变",符合真实成长节奏。

### 14.3 EvolutionLog 数据模型
```json
{
  "date": "2026-01-20",
  "dimension": "social",
  "from": 0.20, "to": 0.35,
  "triggerEvent": "在大理和民宿老板聊了一晚上",
  "userReflection": "原来我也没那么怕和陌生人说话"
}
```

### 14.4 对产品的意义
- **粘性终极来源:** 用户回来不是为了找下一个城市,是为了看看"这次旅行把我变成了什么样"。
- **和情绪地图的关系:** 情绪地图记录"你想去哪"(状态轨迹),人格演化记录"你变成了谁"(成长轨迹)。两条线叠加,构成完整的自我档案。
- **数据闭环:** PersonaScore 不再是输入端的一次性测量,而是贯穿用户生命周期、持续被旅行经历塑造的活数据。

### 14.5 落地优先级
初赛:PersonaScore 静态即可,不做演化。
复赛:实现演化闭环——这是从"推荐工具"升华为"成长伙伴"的关键一步,也是最能打动评审的产品故事。
