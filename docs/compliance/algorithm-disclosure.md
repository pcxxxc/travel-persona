## 1. 推荐算法工作原理

### 1.1 算法概述

旅格 Travel Persona 的推荐算法基于**旅游人格画像匹配**。核心思路是：

1. 构建用户的 16 维旅游人格画像（PersonaScore）
2. 为每个城市构建同维度的城市特征向量（CityScore）
3. 计算用户画像与城市特征的多目标匹配度
4. 综合排序后输出 Top-N 推荐城市及推荐理由

### 1.2 人格画像的 16 个维度

| 类别 | 维度 | 含义 |
|-----|------|------|
| 恢复与放松 | restoration | 恢复感需求 |
| 自然与户外 | nature | 自然环境偏好 |
| 文化与认知 | culture | 文化探索偏好 |
| 美食 | food | 美食体验偏好 |
| 节奏 | pace | 旅行节奏（快/慢） |
| 社交 | social | 社交互动需求 |
| 预算 | budget | 预算敏感度 |
| 审美 | aesthetics | 审美体验偏好 |
| 舒适 | comfort | 舒适度需求 |
| 新奇 | novelty | 新奇体验偏好 |
| 交通 | transit | 交通便利性需求 |
| 低人流 | lowCrowd | 避开人流的偏好 |
| 真实性 | authenticity | 原生态体验偏好 |
| 天气弹性 | weatherFlex | 天气适应性 |
| 预订便利 | bookingEase | 预订便捷度需求 |
| 数字游民 | workation | 远程办公适配度 |

### 1.3 匹配计算流程

推荐生成经过以下管线（Pipeline）：

```
用户输入（问卷/自由文本）
  ↓
Step 1: 人格评分计算（computePersonaScore）
  - 问卷答案 → 加权映射为 16 维评分
  - 自由文本 → LLM 提取增量（可选，熔断降级）
  ↓
Step 2: 约束过滤（constraintFilter）
  - 按硬约束（预算、天数、季节）初筛城市
  ↓
Step 3: 多目标评分（multiObjectiveScorer）
  - 计算用户画像与每个城市的加权匹配度
  - 综合考虑维度权重和偏好强度
  ↓
Step 4: Pareto 优化（paretoOptimizer）
  - 识别 Pareto 最优城市集（无被支配的城市）
  ↓
Step 5: MMR 多样性重排（mmrReranker）
  - 在相关性和多样性之间平衡
  - 避免推荐结果过于同质化
  ↓
Step 6: 可解释性生成（explainability）
  - 为每个推荐城市生成推荐理由
  - 标注最佳匹配维度和需注意的维度
  ↓
输出：PlanResponse（符合 docs/schemas/PlanResponse.json）
```

### 1.4 评分权重

每个维度在匹配计算中有不同权重，反映其对旅行满意度的影响程度：

- 高权重维度（0.11-0.13）：restoration, culture, pace, budget, nature
- 中权重维度（0.07-0.09）：food, social, aesthetics, comfort, transit, lowCrowd
- 低权重维度（0.05-0.06）：novelty, weatherFlex, bookingEase, workation, authenticity

权重设计基于旅行心理学研究，并可根据用户反馈持续校准。

### 1.5 Agent 增强（可选）

当 Agent 增强功能启用时，系统在本地推荐基础上叠加 AI 增强：

- **意图提取**：从用户自由文本中提取更精细的旅行意图
- **解释增强**：为推荐理由增加更自然的语言表达
- **行程调整**：支持用户在行程中实时调整
- **手账总结**：自动总结用户的旅行手账

Agent 增强遵循**无感降级**原则：当 AI 服务不可用时，系统自动回退到本地推荐引擎，用户无感知。

## 2. 人格模型说明

### 2.1 人格画像的构建

用户的人格画像通过两种方式构建：

1. **问卷打分（默认）**：用户回答一组选择题，系统将答案加权映射为 16 维评分（0-1 范围）
2. **自由文本提取（可选）**：用户输入一段描述旅行期望的文字，LLM 提取维度增量并与问卷评分融合

### 2.2 人格校准机制

随着用户使用产品，系统会根据用户的旅行手账和反馈逐步校准人格画像：

- **触发条件**：用户撰写旅行手账并授权分析后
- **校准规则**：
  - 单次校准单维变化不超过 0.08（防止突变）
  - 用户锁定的维度不被自动校准
  - 校准结果以「提案」形式呈现，用户可选择接受或拒绝
- **接受率监控**：系统监控人格更新接受率（`persona_update_acceptance_rate`），过低时触发算法审查

### 2.3 人格画像的保护

- 人格画像数据属于用户个人数据，用户可随时导出和删除
- 人格画像不用于非旅行相关的用途
- 用户可关闭人格校准功能，冻结当前画像

## 3. 数据来源

### 3.1 城市特征数据

城市特征向量（CityScore）的数据来源：

| 数据类型 | 来源 | 更新频率 |
|---------|------|---------|
| 城市 POI | 本地策划的城市数据库 + 百度地图 Place API | 每季度更新 |
| 城市标签 | 编辑团队人工标注 + 数据校验 | 每季度更新 |
| 维度评分 | 基于 POI 统计 + 人工校准 | 每季度更新 |
| 风险标记 | 平台信号（如小红书口碑） + 编辑审核 | 每月更新 |
| 天气数据 | 天气 API 代理 | 实时（7天预报） |

### 3.2 数据质量保障

- **事实校验**：Agent 生成的 POI 必须通过事实校验（与已验证数据源比对），未验证的 POI 会被丢弃
- **数据新鲜率监控**：地图数据新鲜率（`map_freshness_ratio`）作为 SLO 指标，目标 >= 90%
- **数据质量报告**：可通过 `GET /api/v1/ops/data-quality` 查询各维度的数据覆盖率

### 3.3 用户数据

- 用户的人格画像、手账、轨迹数据由用户主动提供
- 用户数据存储在用户自己的数据空间中，不与其他用户共享
- 用户数据不用于训练 AI 模型

## 4. 算法局限性

### 4.1 推荐结果的局限性

1. **主观性**：旅游偏好具有主观性，算法无法保证推荐结果完全符合用户的实际体验
2. **数据覆盖**：城市数据库覆盖的城市数量有限，未覆盖的城市不会被推荐
3. **时效性**：POI 信息（营业状态、价格等）可能因时间推移而过时
4. **上下文理解**：算法对用户意图的理解基于有限的信息输入，可能存在偏差

### 4.2 AI 生成的局限性

1. **准确性**：LLM 生成的内容可能包含事实性错误，尽管经过事实校验，但不保证 100% 准确
2. **一致性**：同一输入在不同时间可能产生略有不同的输出
3. **语言表达**：AI 生成的推荐理由可能偶尔出现不自然的表达

### 4.3 已知的偏差

- 城市数据库以中国境内城市为主，海外城市覆盖较少
- 人格问卷的设计基于一般性旅行心理学，可能无法完全覆盖特定群体的偏好模式
- 推荐算法倾向于推荐数据较完整的城市，数据较少的城市可能被低估

### 4.4 降级行为

当 AI 服务不可用时，系统的降级行为：

| 故障场景 | 降级策略 | 用户感知 |
|---------|---------|---------|
| LLM API 超时 | 使用模板生成推荐理由 | 理由表述更简洁 |
| LLM API 熔断 | 返回空增量，使用问卷评分 | 无感知 |
| Agent Provider 不可用 | 返回本地推荐结果 | 无增强功能 |
| 地图 API 不可用 | 返回本地缓存的 POI 数据 | POI 可能略有滞后 |
| 天气 API 不可用 | 不返回天气信息 | 天气模块空白 |

## 5. 用户控制选项

### 5.1 个性化控制

| 控制选项 | 说明 | 操作位置 |
|---------|------|---------|
| 关闭个性化推荐 | 不使用人格画像，返回通用推荐 | 设置 > 隐私 > 个性化推荐 |
| 重新填写问卷 | 覆盖当前人格画像 | 设置 > 人格 > 重新评估 |
| 锁定维度 | 指定维度不被自动校准 | 设置 > 人格 > 维度锁定 |
| 关闭人格校准 | 冻结当前画像，不根据手账更新 | 设置 > 人格 > 自动校准 |

### 5.2 AI 功能控制

| 控制选项 | 说明 | 操作位置 |
|---------|------|---------|
| 关闭 Agent 增强 | 仅使用本地推荐引擎 | 设置 > AI > Agent 增强 |
| 关闭自由文本提取 | 仅使用问卷打分 | 设置 > AI > 文本理解 |
| 关闭行程润色 | 显示基础行程骨架 | 设置 > AI > 行程润色 |

### 5.3 数据权利控制

| 控制选项 | 说明 | 操作位置 |
|---------|------|---------|
| 导出全部数据 | 下载 JSON 格式的个人数据包 | 设置 > 隐私 > 数据导出 |
| 删除全部数据 | 永久删除账户和所有数据 | 设置 > 隐私 > 删除账户 |
| 关闭手账分析 | 手账内容不进入分析数据池 | 设置 > 隐私 > 手账分析授权 |

### 5.4 功能开关与灰度发布

旅格 Travel Persona 采用功能开关（Feature Flags）管理新功能的灰度发布。用户可能看到部分功能尚未对所有用户开放，这是正常的灰度发布策略。当前管理的功能开关包括：

- `agent_enhancement` — Agent 增强
- `real_time_map` — 实时地图
- `journal_analysis` — 手账分析
- `persona_calibration` — 人格校准
- `multi_city_route` — 多城路线
- `advancedMultiCity` — 多城路线深度优化（付费）
- `exportPersonaReport` — 高级人格报告导出（付费）
- `batchPlanning` — 批量行程规划（付费）

### 5.5 免费层与付费层功能边界

旅格 Travel Persona 采用免费+付费的分层模式，确保基础功能对所有用户可用，高级功能为付费用户提供增值服务。

#### 免费层（始终可用）

| 功能 | 说明 |
|------|------|
| 32 城基础推荐 | 基于人格画像的目的地推荐，覆盖 32 个城市 |
| 三路径比较 | 单城行程的三条候选路线比较与选择 |
| 基础手账 | 旅行手账的创建、编辑与管理 |
| 人格追踪 | 16 维人格画像的展示与变化追踪 |
| 问卷评估 | 标准人格问卷评估与画像生成 |
| 行程保存 | 行程方案的保存与管理 |
| 数据导出 | 个人数据的 JSON 格式导出 |

#### 付费层（需订阅或开通）

| 功能 | 功能开关 | 说明 |
|------|---------|------|
| 多城路线深度优化 | `advancedMultiCity` | 跨城市行程的深度路线编排与优化，支持更复杂的行程约束 |
| 高级人格报告导出 | `exportPersonaReport` | 导出详细人格分析报告（含深度维度解读与旅行建议） |
| 批量行程规划 | `batchPlanning` | 同时规划多个行程方案，支持对比分析 |

### 5.6 商业合作防火墙

旅格 Travel Persona 对推荐结果的公正性做出以下承诺：

1. **POI 排序不受商业合作影响**：目的地推荐和 POI 排序完全基于人格匹配算法，不会因商业合作而改变排序位置。
2. **合作标注义务**：任何与商业实体有合作关系的 POI 或目的地，必须在界面上明确标注"合作"标识，确保用户知情。
3. **算法独立性**：推荐算法的权重、评分逻辑不接受商业定制，所有用户使用相同算法版本。
4. **合作内容隔离**：商业合作内容（如广告、赞助）与算法推荐结果在产品层面严格隔离，互不干扰。
5. **审计可追溯**：合作关系的变更记录留存，可供合规审计查阅。

## 6. 算法审计与持续改进

### 6.1 监控指标

系统持续监控以下与算法相关的指标：

- `plan_generation_time`：推荐生成耗时（SLO: P95 < 2s）
- `api_error_rate`：API 错误率（SLO: < 1%）
- `agent_fallback_rate`：Agent 降级率（目标: < 20%）
- `persona_update_acceptance_rate`：人格更新接受率（目标: >= 40%）

### 6.2 审计机制

- 所有推荐结果保留生成时的算法版本号，便于追溯
- Agent 生成的所有 Patch 经过安全验证和事实校验后才会应用
- 内容安全拦截记录可供合规审查

### 6.3 持续改进

- 根据用户反馈和接受率数据持续优化推荐权重
- 定期更新城市数据库和 POI 信息
- 根据监控指标调整算法参数和降级阈值
