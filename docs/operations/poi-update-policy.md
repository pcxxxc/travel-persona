# POI 数据更新策略

## 概述

本文档定义旅格 Travel Persona 系统中 POI（Point of Interest）数据的更新机制，确保数据长期准确性和合规性。

## 过期阈值

| 指标 | 阈值 | 说明 |
|------|------|------|
| POI 过期 | 180 天 | `lastVerifiedAt` 超过 180 天的 POI 标记为 stale |
| 城市数据过期 | 90 天 | 城市维度向量的新鲜度检查阈值 |
| 无 lastVerifiedAt | 立即标记 | 缺少验证时间的 POI 视为过期 |

## 数据来源优先级

更新 POI 数据时，按以下优先级获取和验证信息：

| 优先级 | 来源 | 可信度 | 说明 |
|--------|------|--------|------|
| P0 | 景区/场馆官方网站 | 最高 | 官方发布的价格、营业时间、地址变更 |
| P1 | 政府文旅数据平台 | 高 | 文旅局、统计局等公开数据 |
| P2 | 地图服务商 API | 中高 | 高德/百度/Google Maps 提供的 POI 信息 |
| P3 | 社交媒体交叉验证 | 中 | 小红书、大众点评等用户反馈 |
| P4 | 用户举报 | 中低 | 用户提交的数据修正请求 |

### 交叉验证规则

1. **P0 优先**：当景区官网与第三方数据冲突时，以官网为准
2. **多源验证**：关键数据（营业状态、价格区间）需至少两个来源确认
3. **时间衰减**：社媒数据超过 30 天视为不可靠，需重新验证
4. **冲突处理**：两来源冲突时，按优先级高的为准并记录冲突日志

## 更新频率要求

### 按城市分级

| 城市类型 | 更新频率 | 示例城市 |
|----------|----------|----------|
| 重点城市 | 每月 | 北京、上海、成都、杭州、西安 |
| 热门旅游城市 | 每月 | 三亚、大理、丽江、桂林 |
| 普通城市 | 每季度 | 芜湖、九江、遵义 |
| 冷门城市 | 半年 | 数据变化少，低频更新即可 |

### 按数据类型

| 数据类型 | 更新频率 | 说明 |
|----------|----------|------|
| 营业状态 | 实时/每周 | 景区关闭/开放影响推荐结果 |
| 门票价格 | 每月 | 旺季/淡季调价 |
| 开放时间 | 每季度 | 季节性调整 |
| 地址/交通 | 半年 | 地铁新开、道路变更 |
| 评分/评价 | 每月 | 影响情报分计算 |

## 去重规则

### 识别重复 POI

1. **名称相似度**：编辑距离 <= 2 且类型相同
2. **坐标距离**：两个 POI 坐标距离 <= 100 米
3. **地址匹配**：地址字符串标准化后完全相同

### 去重策略

- 保留数据更完整的 POI（字段更多的记录）
- 合并来源信息（保留所有可信来源标记）
- 指向被合并 POI 的引用需更新为保留的 POI ID
- 去重日志记录合并原因和操作

## 操作流程

### 日常更新

```
1. 运行 getStalePois() 获取过期 POI 列表
2. 按优先级排序（过期天数最长的优先处理）
3. 逐一通过数据来源更新信息
4. 更新 lastVerifiedAt 为当前时间
5. 运行 validatePOI() 确认数据完整性
6. 记录更新日志
```

### 批量更新

```
1. 从数据来源获取批量数据
2. 按去重规则匹配现有 POI
3. 对匹配的 POI 执行增量更新
4. 对新 POI 执行新增（需 validatePOI 通过）
5. 生成批量更新报告
```

## 质量保障

- 所有更新操作需通过 `validatePOI()` 校验
- 更新后自动检查数据新鲜度
- 生成数据质量报告（`generateQualityReport()`）
- 异常数据标记为 `status: questionable` 待人工审核
