# 0005: 发布基线与版本控制策略

## 状态

已采纳

## 背景

项目进入工程化运营阶段，需要建立明确的 Git 版本控制基线和发布管理规范，确保：
- 代码仓库不包含运行时生成文件、敏感配置、临时产物
- 发布基线可追溯、可回滚
- 多环境部署（开发/生产）配置分离

## 决策

### 1. .gitignore 分层规则

采用分层忽略策略，按类别管理：

| 类别 | 忽略路径 | 原因 |
|------|----------|------|
| 运行时数据 | `.data/` | SQLite 数据库文件，按实例生成 |
| 备份 | `.backups/` | 数据库备份产物，按时间戳命名 |
| 日志 | `.logs/`, `logs/` | 运行时日志，非源码 |
| 环境变量 | `.env`, `.env.*` | 含敏感配置，不纳入版本控制 |
| 依赖 | `node_modules/` | npm 安装产物 |
| 工作区 | `.trae/`, `artifacts/`, `output/` | IDE/工具临时文件 |
| 压缩包 | `*.zip` | 打包产物不入库 |
| Markdown | 根目录 `*.md`，但 `docs/` 除外 | 文档集中在 docs/ 目录管理 |

### 2. 版本号规范

遵循语义化版本 (Semantic Versioning 2.0.0)：
- `MAJOR.MINOR.PATCH`
- MAJOR：不兼容的 API 变更
- MINOR：向后兼容的功能新增
- PATCH：向后兼容的缺陷修复

### 3. 分支策略

- `main`：生产就绪分支，每次合并触发发布基线打标签
- 标签格式：`v{MAJOR}.{MINOR}.{PATCH}`，如 `v0.2.0`
- 每个标签关联一个 Git commit SHA，确保可回滚

### 4. 发布基线检查项

每次发布前需通过：
1. `npm run test:all` 全绿
2. `npm run preflight:production` 通过
3. 数据库备份完成并验证
4. CHANGELOG 更新

## 后果

- 仓库保持干净，仅包含源码和文档
- 任何发布版本可通过 Git 标签精确回溯
- 新环境部署只需 `git clone` + `.env` 配置
- 降低了误提交敏感信息的风险
