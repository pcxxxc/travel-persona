# 旅格产品就绪度审计（2026-07-13）

## 结论

当前阶段：**可公开小规模体验与真实用户验证，尚未达到大规模商业发布条件**。

核心体验已经从“旅行人格展示 Demo”进入“可完成一次真实决策的产品原型”：用户能给出当次取向和现实约束，比较三条路线，保存并删减行程，通过手账形成结构化证据，再由用户确认长期人格变化。智能体只做受约束增强，本地链条始终先产出完整结果。

## 已达到的能力

- 规划：三步输入、预算弹性、必到城市、三条真实可比较的多城方案。
- 路线：茂名到北京 14–21 天继续使用人工校准走廊；其余已覆盖城市之间的 10–21 天长线已能按人格、兴趣、避雷、地理走廊和换酒店成本生成三种往返节奏。
- 通用路线约束：中途城市必须位于出发地与必到城市之间的受控走廊，去程进度单调、返程进度反向单调，三版停留城市数递增；自动回归限制整体绕行比例并阻止广州北上先绕深圳/泉州等常识错误。
- 路线依据：33 条人工跨城连接作为较高把握基线；其他已覆盖城市段使用城市间距离、交通便利度和换乘风险的保守估算，总价仍由停留成本与逐段交通共同计算。用户端会显示估算段数、静态把握和出发日核验要求，不把估算包装成真实车次。
- 出发日核验：百度跨城公交 Provider 已支持日期、时段、少换乘/时间短/低价策略以及火车优先，并解析车次、票价、到发站；无 Key、无日期或请求失败时保持静态方案。
- 推荐去偏：单城评分已统一使用固定 32 城标准记录库，并用城市情报中的拥挤、预约和天气分数替代粗糙二元标签；7 类人工青年场景保持动机差异。1,296 组组合压测产生 16 个不同 Top 1、前三覆盖 24 城；占比最高的武汉为第一名 22.7%、进入前三 56.9%，均低于自动门禁上限。
- 路线可比较性：三套多城节奏在点击前同屏展示总价区间、换城段数、机动天数和预算状态；“更低成本”按保守最高估算排序，不用诱人的最低价误导选择。
- 地图可读性：长线路线会等容器尺寸稳定后再适配范围；手机地图高度提升到 340px 并进入可读缩放，地图下方同步显示当前方案的完整城市顺序。容器尺寸观察器覆盖分屏、旋转和桌面窗口变化，页面重绘会清理旧地图与监听。
- 默认选择：人格本选的匹配优势明显且费用不超过硬上限时会默认选中；优势不明确、超预算或多城路线时继续默认现实平衡。
- 人格：16 个主维度、证据可靠度、单次变化上限、用户接受/拒绝、锁定与排除证据。
- 成长：手账原文与人格证据分离，敏感记录不进入人格分析；一条记录最多提出两项变化。
- 手账自主权：每条记录都能主动加入或退出分析，并可二次确认后删除；停止分析和删除都会同步清理失效的待确认变化，路线解释也不会留下孤立引用。
- 取舍证据：删城会形成可撤销的行为记录，但删除本身永远不等于“不喜欢”；只有用户主动解释并授权后才进入证据池，撤销路线会保留手账并停止关联证据。
- 成长可见性：用户端会按时间展示计划、已授权证据和亲自确认的变化，不复述手账原文或敏感推断。
- 成长可解释性：“我的旅格”区分最近一次规划信号与已确认长期倾向；长期倾向逐项显示当前位置、证据数量和把握度，尚未达到完整人格门槛时明确说明原因，不再只显示抽象标签或内部小数。
- 提案可审计性：每项候选变化展示支持/反例数量、变化后可能范围和仍需的数据；反例权重不低于支持证据时不会生成提案。
- 提案连续性：同一维度不会并列出现多个重复提案；新复盘会合并全部已授权证据，旧提案改为已取代审计记录，服务启动时也会迁移历史重复数据。
- 提案有效性：待确认变化会与当前有效证据对账；撤回分析授权、删除手账或撤销关联路线取舍后，失去依据的提案会立即失效。
- 已确认判断的证据口径：用户已亲自确认的长期判断不会因来源撤回而被系统静默反转，但成熟度只计算仍有效的证据；页面会明确标注“来源证据已撤回，由你确认保留”，不再把历史证据伪装成当前有效证据。
- 人格可纠正性：从第一个已确认维度开始，用户即可主动复核为“仍然准确、只代表当次旅行、我已经变化”；重新定位会保留旧值与新值，并再次等待用户确认，不由系统静默覆盖。
- 复盘可靠度：普通记录、计划取舍、完整旅后复盘分别采用 0.35、0.40、0.90 的先验；前两类单条不能提出长期变化，完整复盘必须关联真实已完成行程并填写三段结构化摘要。
- 完整复盘过程：用户分别回答“这趟是否值得、真正留下了什么、计划与实际差异”，再用自己的话收尾；已完成行程冻结原计划，实际偏差进入复盘而不是反向改写历史计划。
- 旅行实况：开始旅行后原计划冻结，用户可逐城标记实际到访或没去成、以半天为单位调整真实停留，并补记/撤销临时城市；四类事件均为结构化、可追溯、当前用户所有。
- 实况隐私门禁：实况只用于计划对照和真实足迹，不会单独进入人格证据池；完整复盘会显示“计划 / 到访 / 跳过 / 新增”摘要，但是否授权分析仍由用户再次选择。
- 到访真实性：完成状态不再等同于到访全部计划城市；有实况时只统计实际到访与临时新增，无实况的历史完成行程进入待确认集合，不污染到访地图和人格统计。
- 行程删除：用户可二次确认后删除单条计划及实况；关联手账保留并解除 tripId，不因删除路线而丢失个人记录。
- 行程对账：刷新后以服务端记录为权威，远端已删除记录不会被本地旧副本复活；创建/更新失败会进入明确待同步状态，早期仅存本设备的计划显示迁移入口，不自动上传或静默丢弃。
- 保存反馈一致性：首次保存成功后立即重绘行程列表，清除短暂的“等待同步”；保存期间按钮锁定为“保存中”，阻止慢网络下的重复行程。失败时同样重绘为真实待同步状态并保留本地方案。浏览器实测尽量多城版的 11 节点快照能完成保存，随后已清理测试记录。
- 成长进入推荐：下一次计划只使用服务端签名会话下、经用户确认且有证据计数的长期维度；客户端自带画像不会进入算法，避免篡改或跨用户污染。
- 个性化边界：关闭个性化或长期记忆后，即使服务端仍有历史档案也会立即切换为非人格计算；推荐结果同步显示“已采用的确认维度数 / 冷启动 / 非个性化”状态。
- 韧性：本地规划先完成，Agent 只允许修改白名单字段；超时、熔断和未配置均无感保留本地结果。
- 降级可观测：用户端规划成功、API 错误、Agent 降级、地图快照和运行时错误采用匿名白名单事件上报；运营接口可按 15 分钟和 24 小时汇总，禁止手账、城市、路线、人格内容、错误原文、堆栈和用户/行程标识进入监控标签。
- 监控留存：匿名指标写入 SQLite 独立表，默认保留 30 天并限制单指标最多 10000 个点；服务重启后可恢复，写入异常时自动转为内存统计并把健康状态标为降级，数据库备份清单同步记录指标行数。
- 预算可信度：当预计区间跨过用户最高上限时，会显示具体风险金额和压缩动作；明显超支方案仍由硬上限淘汰。
- 行程真实性：无日期行程只能继续规划，详情页可安排或调整出发日并自动计算结束日；到出发日后才能进入实况，结束日到达后才能标记完成。前后端双重门禁阻止无日期、未来出发或尚未结束的状态污染人格与到访数据。
- 错误体验：前端不再使用原生弹窗或直接显示服务端报错；统一轻提示支持无障碍播报，并在手机端避让底部导航。
- Web 安全：生产写接口要求请求来源命中正式域名白名单，API 响应禁止缓存；页面已启用无内联脚本的 CSP、HSTS、禁止嵌入、MIME 嗅探防护与跨窗口隔离，地图瓦片只开放明确的图片域名。
- 工程：SQLite 持久化、签名访客会话、正式身份 Provider 合同、游客数据幂等继承、跨用户隔离、数据导出/删除、生产预检与全量自动测试。
- 内容安全：已经具备可配置语义 Provider、超时、三次失败熔断、本地脱敏和隐私授权过滤；Provider 不可用时不阻断正常旅行链路，受限手账仍不会进入人格或 Agent 上下文。
- 恢复能力：真实 SQLite 在线备份会生成 SHA-256 清单并执行 `quick_check`；数据库卷与备份卷分离，恢复要求停机和显式确认，并在覆盖前保留原数据库。
- 接口收口：生产默认只开放 `/api/v1`，旧版接口必须显式开启，生产预检会拒绝开启旧接口的配置。
- 发布完整性：城市种子数据已从历史站点迁入 `src/`；隔离发布测试会在没有 `public-site/` 的环境中重新加载城市并生成路线。
- 运营可信度：数据质量接口已改用真实 `traitVector` 字段，16 维覆盖率和地区分布不再误报，并纳入跨城连接数量、平均置信度和待实时核验数量。
- 视觉：已形成克制、抽象、偏编辑设计的用户端，不再依赖具象卡通人物。

## 关键边界

### 数据覆盖

当前正式推荐库固定为 32 个目的地、147 个 POI，平均每城约 4.6 个 POI；跨城图有 33 条人工静态连接，其余已覆盖城市段可生成明确标注的距离估算。数据结构完整度为 A；按 32 城、每城 20 个 POI、80 条已核验跨城连接的首发产品目标衡量，当前就绪度为 72/B。现阶段不继续扩城，优先补齐这 32 城的 POI 深度、连接密度、季节波动和实时事实。未覆盖的出发地或必到城市不会被宣称拥有完整多城能力，也不会把重复城市或未经核验的浅数据灌入覆盖统计。

### 地图与实时事实

百度地图 Provider、超时、缓存和离线降级已经存在，主用户链会在本地方案显示后后台核验城市与最多 12 个关键地点；BD-09 坐标会转换为 WGS84 后再用于当前底图。填写出发日期后还会核验最多 10 段跨城公共交通。默认仍是本地快照与 OpenStreetMap 展示；当前环境未配置百度 Key，因此实际在线跨城返回仍未完成真实调用验收，营业时间、余票、最终票价、门票与预约变化不能被视为实时事实。

### 身份与商业化

签名访客身份适合公开体验和小规模内测。当前已经实现可替换的 Bearer 身份验证合同、不可逆本地账号标识、游客数据一次性幂等迁移，以及人格与隐私的保守合并规则；身份服务异常时不会把账号请求误降级为游客。当前环境尚未接入真实手机号、微信或 OAuth Provider，因此跨设备同步、付费、正式会员与客服追责仍未完成真实联调和验收。

### 内容安全与合规

本地词库和隐私分级是第一层防护，外部语义 Provider 的调用契约、超时、熔断与隐私化传输边界已经实现；当前环境仍处于本地模式，尚未对真实服务完成准确率、误判、延迟与数据处理协议验收。大规模公开发布还需要上线真实语义服务、误判申诉、审计留痕、规则热更新、数据保留策略与隐私影响评估。平台口碑数据必须遵守来源平台条款和授权边界，不能把无授权爬取当作长期数据源。

### 工程仓库

`public-app/`、`src/`、`test/`、`docs/` 是当前主产品；`public-site/`、`travel7.9/`、`travel-persona*/` 与多个 zip 是历史版本和交付包。它们占用大量空间并增加误发布风险。当前根目录没有 Git 仓库，缺少可靠的版本追踪与发布回滚基线。

Docker 发布面已收紧为显式复制 `server.js`、`src/` 和 `public-app/`，历史目录、测试与审计资产不会进入镜像。发布面隔离测试已证明推荐链不再依赖未打包的历史站点；由于本机没有 Docker，容器镜像本身仍需在 CI 或有 Docker 的环境中完成构建验证。

数据库恢复链已从模拟记录升级为真实 SQLite 在线备份：`/app/.data` 与 `/app/backups` 使用独立 Docker 卷，备份带 SHA-256 清单、SQLite 完整性校验、保留数量/天数策略，恢复会先验证并保留覆盖前数据库。应用内原有用户/系统/人格备份仅作为逻辑快照，不计入灾难恢复能力。当前已通过自动化备份与恢复测试，并对本地实际数据库完成一次 24 条持久化记录的备份、恢复与复验；全量测试前后工作数据库均保持 24 条不变。仍需在真实部署存储上安排定时任务并完成一次停机恢复演练。

## 优先完善方向

### P0：把“小规模体验”变成“可控公测”

1. 将已实现的身份合同和游客数据迁移接入正式手机号、微信或 OAuth Provider，完成多设备、退出登录、令牌过期与账号合并验收。
2. 配置百度地图，建立 POI 事实校验、营业时间与路线矩阵缓存；对过期数据明确降权。
3. 把已实现的语义内容安全接口接入正式服务，完成误判评测、审计日志、隐私政策、用户协议与删除时限核验。
4. 单实例匿名监控已持久化；多实例放量前接入外部时序库，并建立 CI、告警和版本回滚。为独立备份卷配置外部定时任务，并在真实部署环境完成恢复演练。

### P1：把 32 城首发库做深

1. 已为 32 个覆盖城市建立通用往返走廊算法，并保留 33 条人工连接基线；下一步将距离估算逐段替换为地图/票务 Provider 的票价、耗时、班次密度、季节波动和换乘风险。
2. 冻结 32 城范围；每城逐步维护 20-40 个经过核验的 POI、住宿区域、预约规则、季节差异和预算波动，不以增加城市数量替代数据深度。
3. 建立来源、更新时间、置信度和相互印证字段；口碑信号必须能追溯到来源批次。
4. 已把“删除计划城市”接入可撤销的取舍记录与主动解释链路，删除动作本身仍不会成为人格证据。
5. 已补齐实际到访、跳过、临时新增和停留偏差的结构化实况层；下一步在真实用户样本中评估哪些偏差值得进入完整复盘提示，继续避免自动人格化。
6. 7 类人工场景和 1,296 组组合压测已进入自动门禁；下一步继续加入路线合理性、预算误差、预约冲突和真实用户采纳率，不能只看模型分数。

### P1：把旅格成长做成长期留存核心

1. 已增加“计划时的我 / 实际发生 / 我确认的变化”三段时间线，并明确隐藏手账原文。
2. 已为每次提案增加反例和可能范围，并在反例权重过高时停止提案；后续应继续校准区间覆盖率。
3. 已将已确认长期维度作为下一次推荐的服务端可信输入，并在结果页显示采用数量；下一步用真实留存数据评估它是否提高路线采纳率，而不是只提高解释感。
4. 继续区分长期倾向、当次状态和现实妥协；预算紧不应被误判为永久偏好低成本。
5. 复盘重点围绕“更明晰自己向往之所”，不要把旅行包装成必须成长的任务。

### P2：商业转化

1. 支持可分享行程、同行协作、日历与离线清单导出。
2. 在用户确认路线后再提供酒店、交通或体验预订，不让商业排序污染人格推荐。
3. 商业合作必须显示利益关系，并保留“最符合人格”和“更低成本”两个独立选项。

## 暂时不要继续做

- 不再增加更多人格名称或视觉皮肤，现阶段瓶颈不是人格数量。
- 不再为单个路线继续硬编码更多城市，先建设可泛化交通图和数据管线。
- 不再扩大“看起来智能”的长解释，先提高事实覆盖、来源可信度和误差可见性。
- 不在未建立授权与合规链路前批量抓取社交平台内容。

## 上线门槛

公开小规模体验：当前能力基本满足，前提是使用 HTTPS、生产密钥、限流、真实定时备份，并明确体验版数据边界。当前自动化测试证明恢复逻辑可用，但不能替代部署环境中的恢复演练。

大规模商业发布：必须完成账号、地图实时数据、语义内容安全、32 城数据深挖、持续监控与恢复演练后再放量。
